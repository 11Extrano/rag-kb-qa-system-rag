## Purpose

接受管理员上传文档，对文档执行文本清洗与可配置的拆分（可选按标题/结构 + 长度与重叠），产出文本片段并写入片段表，供向量存储与检索使用。

## Requirements

### Requirement: 系统接受管理员上传文档

系统必须接受由系统管理员发起的文档上传，持久化原始文档（或从 PDF 中提取的文本），并发出或记录「文档上传完成」事件。支持的文件类型包括 .txt、.md、.html 与 .pdf；对 .pdf 须先提取文本再持久化为文档内容。（MUST）

#### Scenario: 管理员上传单个文档

- **WHEN** 系统管理员通过指定上传接口提交文件
- **THEN** 系统必须将文档持久化并记录元数据（如 doc_id、文件名、上传时间、状态）至文档表，并将该事件视为「文档上传完成」

#### Scenario: 拒绝不支持的文件格式

- **WHEN** 系统管理员提交了不支持的文件类型
- **THEN** 系统必须拒绝上传，并返回明确的不支持格式错误信息

#### Scenario: 管理员上传 PDF 时系统提取文本并走后续流水线

- **WHEN** 系统管理员提交了 .pdf 文件
- **THEN** 系统必须从 PDF 中提取文本，将提取出的文本作为文档内容持久化至文档表，并视为「文档上传完成」，后续对该文档执行清洗、拆分与向量化与其它支持格式一致

#### Scenario: PDF 提取结果为空或无效时记录失败且不继续拆分

- **WHEN** 对上传的 .pdf 执行文本提取后，提取结果为空或仅包含无效/空白字符
- **THEN** 系统必须记录该文档为失败状态，且不得对该文档执行拆分或向量化（或必须向管理员返回明确错误信息）

---

### Requirement: 系统对上传文本进行清洗

系统必须对已存储的文档内容执行文本清洗（如规范化空白、去除控制字符、可选编码处理），并记录或发出「文本清洗完成」。（MUST）

#### Scenario: 上传后执行清洗

- **WHEN** 文档已上传并存储
- **THEN** 系统必须对其内容执行文本清洗，并标记「文本清洗完成」

#### Scenario: 清洗后文本为空或无效

- **WHEN** 清洗后提取的文本为空或无效
- **THEN** 系统必须记录失败，且不得对该文档继续执行拆分（或必须向管理员返回错误）

---

### Requirement: 系统将清洗后文本拆分为片段

系统必须按可配置的两段策略将清洗后的文本拆分为文本片段（chunk），并写入片段表、供向量存储使用，记录或发出「文本拆分完成」。（MUST）

#### Scenario: 由清洗后文本产出片段

- **WHEN** 某文档的文本清洗已完成
- **THEN** 系统必须将文本拆分为片段，将片段（chunk_id、doc_id、原文 text、可选 metadata）持久化至片段表，标记「文本拆分完成」，并向下游提供片段列表

#### Scenario: 按 Markdown 标题切分（可选第一段）

- **WHEN** 配置中启用了「按标题」切分且文档为 Markdown（如 #、##）
- **THEN** 系统必须先按 Markdown 标题/小节切分为大块，每块可带标题作为 metadata；再对每块按第二段规则切分；若无明显 Markdown 结构则整篇进入第二段

#### Scenario: 片段大小与重叠可配置

- **WHEN** 配置中指定了是否按标题切分、最大长度、重叠大小（如字符数或 token 数）
- **THEN** 系统必须按该参数产出片段，第二段在换行/句号等边界切、单块不超过最大长度、块间保留重叠（如约 10%～20%），避免句意在边界被截断
