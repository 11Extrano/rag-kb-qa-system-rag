## Purpose

接收用户检索请求，将查询向量化后在向量库做 top-k 相似度检索，并根据 chunk_id 从 MySQL 取原文与文档元数据，组装为「匹配到的内容」列表供答案生成使用。

## Requirements

### Requirement: 系统接受用户的检索请求

系统必须接受检索请求（如用户问题或查询字符串），并记录或视为「检索请求接收」，请求可归属到用户角色。（MUST）

#### Scenario: 用户以问题发起检索请求

- **WHEN** 用户通过指定接口（如 API 或 UI）提交问题或查询
- **THEN** 系统必须将其作为检索请求接受，并记录「检索请求接收」

#### Scenario: 查询为空或缺失

- **WHEN** 检索请求的查询为空或缺失
- **THEN** 系统必须拒绝该请求并返回校验错误

---

### Requirement: 系统按向量相似度匹配相关文档

系统必须将用户查询转为嵌入（通过与向量存储相同的 EmbeddingProvider），在 LanceDB 执行 top-k 相似度检索得到 chunk_id 与 score；再根据 chunk_id 从 MySQL 片段表取原文、关联文档表取文件名等，组装为「匹配到的内容」列表，并记录或发出「相关文档匹配完成」。（MUST）

#### Scenario: 匹配返回带原文与来源的内容列表

- **WHEN** 已收到有效检索请求且向量库中已索引片段
- **THEN** 系统必须计算查询嵌入、调用向量存储 top-k 检索得到 [{ chunk_id, score }, ...]，再根据 chunk_id 从 MySQL 取原文与文档元数据，组装为「匹配到的内容」列表（含 chunk_id、doc_id、原文 text、score、文档名等），按 score 降序，并标记「相关文档匹配完成」

#### Scenario: 向量库中无片段

- **WHEN** 向量库中没有任何片段（如尚未索引任何文档）
- **THEN** 系统必须返回空匹配列表，并可返回知识库为空的提示
