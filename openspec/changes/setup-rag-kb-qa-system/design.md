## 背景

本方案面向基于自有文档构建知识库并支持自然语言问答的 RAG（检索增强生成）系统，采用领域驱动设计（DDD）划分界限上下文：文档处理、向量存储、检索匹配、答案生成。领域事件与角色已通过事件风暴确定：系统管理员负责文档上传与知识库管理，用户发起检索并获取基于文档内容的答案。

当前为从零建设阶段，无既有代码或历史约束。技术栈已确定：前端采用 React 18 + TypeScript，构建工具为 Rsbuild；后端采用 Egg.js + TypeScript，数据层使用 MySQL；向量存储采用 LanceDB（Node 嵌入式）。上述选型作为本设计文档的既定前提。

## 目标与非目标

**目标：**

- 实现端到端 RAG 流水线：文档上传 → 清洗/拆分 → 向量化存储 → 检索匹配 → 答案生成。
- 明确各模块职责与接口，便于分阶段实现与测试。
- 支持系统管理员上传与管理文档、用户发起检索并获取带引用的答案。

**非目标：**

- 多租户、细粒度权限、审计日志（可后续迭代）。
- 非文本格式（如图片、表格 OCR）的深度处理；本次以文本为主。

## 关键决策

1. **模块划分与依赖方向**  
   - 按界限上下文拆为四个逻辑模块：文档处理、向量存储、检索匹配、答案生成。  
   - 依赖方向：文档处理 → 向量存储 → 检索匹配 → 答案生成（与事件风暴一致）。  
   - **理由**：与领域模型一致，便于后续微服务或单体内包边界划分。其他方案（如“检索+生成合并”）会模糊职责，不利于单独扩展检索或生成策略。

2. **技术栈**  
   - **前端**：React 18 + TypeScript，构建工具 Rsbuild。  
   - **后端**：Egg.js + TypeScript，关系型数据 MySQL（文档元数据、文本片段等可落库；向量仍由向量库存储）。  
   - **理由**：统一 TypeScript 便于类型共享与协作；Rsbuild 与 Egg 与现有生态兼容，利于首版交付与后续迭代。

3. **文本分块策略（组合思路）**  
   - **第一段：按结构切（可选）**  
     - 若文档为 Markdown 或含标题（如 `#`、`##`、`<h1>`～`<h6>`），先按标题/小节切分为若干「大块」，每块可带标题作为 metadata，便于检索时保留上下文。  
     - 若无明显结构，跳过此步，整篇进入第二段。  
   - **第二段：按段落/句子 + 长度 + 重叠**  
     - 对第一段得到的每一块（或整篇）再切：优先在换行、句号等边界切，单块不超过配置的**最大长度**（如 500 字或 256～512 token），块与块之间保留**重叠**（如 50 字或约 10%～20%），避免句意在边界被截断。  
   - **可配置项**：是否启用「按标题」切分、最大长度、重叠大小；实现时可用递归分隔符（先 `\n\n`，再 `\n`，再空格）配合长度与 overlap。  
   - **理由**：组合比单用固定长度更少切断句意，比单用标题更能控制单块大小；成熟且易调优。

4. **文档与 Chunk 的存储**  
   - **MySQL 存**：  
     - 文档表：doc_id、文件名、上传时间、状态等元数据。  
     - 片段表：chunk_id、doc_id、原文 text、可选 metadata。  
   - **LanceDB（向量库）存**：  
     - 每条记录 = 向量 + chunk_id（不存原文），用于按相似度检索。  
   - **使用方式**：检索时先在向量库用 query 向量查出 top-k 的 chunk_id，再用 chunk_id 到 MySQL 查片段表取原文、并关联文档表取文件名等，供 LLM 生成答案与前端展示引用。  
   - **理由**：向量库只做相似度查找，原文与来源集中在 MySQL，避免重复存储、也便于管理和展示。

5. **向量存储选型**  
   - **选用 LanceDB**：Node 嵌入式，与 Egg 同进程；通过 `vectordb`（LanceDB Node 包）调用，数据持久化到本地目录，无需 Docker 或独立向量服务。若后续需分布式或更大规模，可再评估 Weaviate、Qdrant 等并替换实现。  
   - **理由**：纯 Node/TypeScript 栈、无额外进程与运行时，部署简单；与 MySQL（存文档与 chunk 原文）分工清晰。

6. **嵌入与 LLM 调用方式**  
   - 嵌入：通过统一抽象（如 `EmbeddingProvider`）调用本地模型或远程 API（OpenAI-compatible / 本地 embedding 服务）。  
   - 答案生成：通过统一抽象（如 `LLMProvider`）调用本地或远程 LLM，便于后续替换模型。  
   - **理由**：抽象可隔离厂商与部署方式，便于测试与迁移。

7. **向量匹配结果的确定**  
   - 向量检索输出：按用户 query 向量在 LanceDB 中做相似度检索，取 **top-k**（k 可配置，如 5～10）条，得到列表 `[{ chunk_id, score }, ...]`，可按 score 排序或过滤（如低于某阈值则丢弃）。  
   - 内容确定：用上述 chunk_id 列表到 MySQL 查片段表取原文、关联文档表取文件名等，得到「匹配到的内容」列表，每项含 chunk_id、doc_id、原文 text、相似度 score、文档名等，按 score 降序交给答案生成模块。  
   - **理由**：只把“最相关的 k 段”交给 LLM，控制 token 与噪声；分数与来源保留，便于引用与后续调优。

8. **检索结果与 LLM 的交互格式与提示词**  
   - **传给 LLM 的数据**：  
     - **System**：固定约束，例如“你是一个知识库助手。请仅根据下面【参考内容】回答用户问题；若参考内容中无法得到答案，请说明并不要编造。”  
     - **User（或单条 prompt）**：包含【参考内容】+ 用户问题。参考内容 = 将上述「匹配到的内容」列表格式化为可读文本（如每段标注来源 doc/chunk），再拼接；最后附上“用户问题：xxx”。  
   - **为什么**：RAG 依赖“依据检索到的文本回答”，必须把检索结果显式放进 prompt，并约束“仅根据这些内容回答”，否则模型易用自身知识产生幻觉；把参考内容与问题分开写，便于模型区分依据与问题，也便于做引用与审计。

9. **接口形态**  
   - 首版采用 REST API：管理员侧上传/列表/删除文档；用户侧问答接口（请求 question，返回 answer + 引用片段）。  
   - **理由**：REST 易于对接前端或其它系统；若后续需要流式输出，可在同一 API 上扩展 streaming。

## 风险与取舍

- **调用外部接口可能慢或失败** → 嵌入接口、LLM 接口都可能超时或暂时不可用。应对：上传与检索接口加超时与重试；答案生成失败时做兜底（例如只把检索到的片段返回给用户，而不是报错或白屏）。
- **Chunk 策略不当导致检索噪声** → 在文档处理模块中可配置 chunk 大小与重叠，并在设计/任务中预留“评估与调优”步骤。
- **采用单体服务** → 文档处理、向量存储、检索匹配、答案生成均在一个 Egg 应用内；部署与运维简单，模块边界清晰，后续若有需要再拆分为多服务。

## 迁移与部署

- 无既有系统，无迁移。部署步骤：先部署向量库与（若使用）嵌入/LLM 服务，再部署应用；配置中指定嵌入模型、向量库连接、LLM 端点。

## 开放问题

- 嵌入模型与 LLM 的具体选型（本地 vs 云、模型名称）可在实现阶段按环境确定。
- 是否需要流式输出答案（如 SSE），可在首个可运行版本后再加。
