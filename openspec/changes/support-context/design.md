## Context

- 现状：`retrievalMatch.retrieve(question)` 仅对单字符串做 embed；`answerGeneration` 使用 system + 单条 user（参考 + 问题）；`qa` 无会话模型。
- 约束：Egg + MySQL + LanceDB；不做用户账号；设计文档已定 **方案 C**（改写与作答分请求、分 prompt，可共用 `llmProvider`）。
- 参考：`docs/多轮知识库对话-上下文设计说明.md`。

## Goals / Non-Goals

**Goals:**

- 持久化会话与 user/assistant 消息，按 `conversationId` 拉取历史。
- 检索链路：**改写**（输入含 assistant，输出一句检索 query）→ **embed** → 与现有一致的 chunk 组装与 score 排序。
- 作答链路：system + **裁剪后的多轮历史** + 本轮参考 + 用户问题；**token 预算** + 文档中的裁剪顺序。
- HTTP/SSE 契约扩展 `conversationId`（及会话创建策略），前后端可测。

**Non-Goals:**

- 用户登录、权限、多租户隔离（非本会话 id 维度）。
- **多会话历史与会话列表**：不要求客户端同时保留/切换多套会话记录，不要求「历史会话列表」UI，不要求 `GET /api/...` 枚举用户全部会话；**新会话**仅指换新 `conversationId` 并清空当前界面（见 tasks）。服务端仍按 `conversationId` 存**单线程**多轮消息，旧 id 数据可留库但不作为本期产品能力暴露。
- 对久远历史的自动摘要注入（可作为后续 change）。
- 改写失败策略以外的复杂降级（可在 tasks 中实现「回退当前句」作为最小兜底）。

## Decisions

1. **会话 id 与首写库时机**（见下文「已决议」）  
   - 由**客户端生成** `conversationId`（建议 UUID v4），随每次问答请求上报。  
   - **服务端在首次收到**带该 id 的问答请求时**隐式创建**会话行（若不存在则 INSERT），并写入本轮消息；不要求先调「创建会话」接口。

2. **改写输入窗口**  
   - 最近 N 轮（含 user/assistant），N 可配置；改写 prompt 与作答 prompt **严格分离**（不同 system 指令）。

3. **Embedding 输入**  
   - **仅**改写输出的那一句；不把整段对话拼起来 embed（与方案 B 区分）。

4. **作答侧历史**  
   - 与改写窗口可同 N 或略大/略小；二者均受 token 预算约束后再送入 LLM。

5. **Token 计数**  
   - 使用与模型一致的 tokenizer 或近似（如按字符/按现有库）；在 spec 中要求「可配置上限 + 裁剪顺序」，实现可选用 `tiktoken` 同类方案或项目已有工具。

6. **与现有一轮兼容**  
   - 无 `conversationId` 或无历史时：改写输入仅为当前 user 句（或与「仅当前句」等价），行为接近现网单轮。

## Risks / Trade-offs

- **[Risk] 延迟与费用上升**（每轮多一次改写 LLM）→ 可缓存无关、监控 p95；必要时后续做仅 user 的快捷路径（产品决策）。  
- **[Risk] 改写质量影响检索** → 兜底：改写失败或空结果时用本轮用户原文做 embed；日志可观测。  
- **[Risk] token 实现与模型不一致** → 预留余量（安全 margin），并在配置中可调。  
- **[Trade-off] 表结构与迁移** → 新建表需迁移脚本与回滚说明。

## Migration Plan

1. 新增表并部署（会话、消息）；旧数据无会话，不影响。
2. 部署后端：新 API 字段可选时先兼容旧客户端（无 `conversationId` 走单轮路径）。
3. 前端升级后默认带 `conversationId`。
4. 回滚：保留表，代码回退到旧控制器路径；或 feature flag 关闭改写与历史。

## 已决议（原 Open Questions）

1. **会话由谁创建、是否首条消息隐式创建**  
   - **`conversationId` 由客户端生成**（UUID），服务端不预分配。  
   - **首条带该 id 的问答请求**到达时，若库中无此会话，则**隐式创建**会话并落库消息；后续轮次复用同一 id。  
   - 不提供单独的「先 POST 空会话再提问」作为必走路径（若产品日后要「会话标题」等，可再增加可选 `POST /api/conversations`）。

2. **N 与 token 上限默认值（dev / prod）**  
   - **历史轮数 N**（参与改写与作答窗口，按「user+assistant」为 1 轮）：默认 **10 轮**；生产环境可通过配置改为 **6～8 轮** 以控 token，**开发环境**保持默认 10 便于调试。  
   - **单次生成输入 token 预算**（system + 历史 + 参考 + 用户问题，计数后裁剪）：默认 **8000 tokens**（含安全余量前的目标上限）；**生产**建议环境变量设为 **6000～8000** 视模型与成本调整，**开发**可用 **8000** 或略高以便排查截断问题。  
   - **输出预留**：从模型上下文上限中预留 **1024～2048 tokens** 给回答（与现有 `llmProvider` 配置对齐），剩余部分计入「输入预算」计算。  
   - 以上均为默认建议，均以 **配置文件 / 环境变量** 可覆盖为准。

3. **会话列表 / 多会话历史（本期不做）**  
   - **不包含**：枚举全部会话、会话列表 UI、客户端维护多套并行会话历史。  
   - **仍需要（单线程）**：按**当前** `conversationId` 查询该会话下消息，供**服务端**拉取多轮上下文做改写/作答；若产品要做**整页刷新后还原当前这一串对话**，可用同一只读接口，**不是**多会话管理。  
   - **新开话题**：客户端换新 `conversationId` 并清空界面即可；旧 id 在库中可不删、不在本期界面呈现。  
   - **后续**若要做历史会话列表或删除，另起 change。
