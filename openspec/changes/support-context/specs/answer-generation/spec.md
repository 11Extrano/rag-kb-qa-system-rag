## ADDED Requirements

### Requirement: 系统对生成答案的输入执行 token 预算与裁剪

系统必须在调用 LLM 生成答案（含流式与非流式）之前，对即将发送的完整输入（含 system、多轮历史、参考内容、用户问题等）按配置的 **token 预算** 进行计数；若超出预算，必须按以下顺序循环裁剪直至落入预算或达到实现约定的下限：① 在多条检索参考中移除 **score 最低**的一条；② **成对**移除最早一轮 user 与 assistant 消息；③ 对本轮用户输入仍过长时进行截断或摘要；④ 最后方可缩短 system 中非关键语句。轮数上限可与 token 预算同时使用。（MUST）

#### Scenario: 超长输入经裁剪后仍可生成

- **WHEN** 组装的完整输入超过 token 预算
- **THEN** 系统必须按上述顺序裁剪后再调用 LLM，且不得静默截断单条消息中间导致严重语义残缺作为主要手段（与 design 中「避免硬切单条」一致）

## MODIFIED Requirements

### Requirement: 系统根据检索片段与用户问题生成答案

系统必须通过统一抽象（如 LLMProvider）调用配置的 LLM，根据用户问题与「匹配到的内容」列表（含原文、score、文档名等）生成自然语言答案，记录或发出「答案生成完成」，并将答案返回给用户。当存在会话历史时，传给 LLM 的 messages 须在 system 与本轮参考内容之外，包含经 **token 预算与裁剪**（见 ADDED Requirement）处理后的多轮 user/assistant 历史；在无会话历史时，行为可与原单轮一致。User 侧须包含【参考内容】（匹配内容格式化并标注 doc/chunk 来源）与本轮用户问题。System 约束仍须包含「仅根据【参考内容】回答，若无法得到答案则说明不编造」。（MUST）

#### Scenario: 返回带内容的答案

- **WHEN** 用户已提交问题且检索上下文返回了至少一个匹配片段
- **THEN** 系统必须将参考内容、本轮用户问题及（若存在）裁剪后的历史消息按约定格式组装为 prompt，调用 LLMProvider 生成答案并返回给用户，并标记「答案生成完成」

#### Scenario: 无匹配片段时的应答

- **WHEN** 检索上下文返回空片段列表
- **THEN** 系统仍须响应用户（如提示未找到相关内容或知识库为空），且不得调用 LLM 在无依据情况下编造内容

#### Scenario: 多轮历史存在时答案仍仅依据参考内容

- **WHEN** 会话中存在历史消息且本轮检索返回至少一个匹配片段
- **THEN** 系统必须在 prompt 中保留 system 约束，使模型仅依据本轮【参考内容】作答，历史仅用于理解指代与连贯性，不得用历史替代参考块中的事实依据

---

### Requirement: 系统可支持流式生成并返回答案

在保留一次性返回答案能力的前提下，系统必须支持「流式生成并返回答案」的可选路径：通过 LlmProvider 的流式调用（如 `stream: true`）逐段获取 LLM 输出，并将答案以流式形式写出；引用在流结束后随元数据或单独事件返回。流式路径下所使用的 prompt 约束（含多轮历史、参考内容、token 裁剪）必须与一次性生成一致。（MUST）

#### Scenario: 流式生成有匹配时的答案

- **WHEN** 调用方请求流式生成且检索返回至少一个匹配片段
- **THEN** 系统必须按与一次性生成相同的 prompt 约束（仅根据参考内容回答、含历史时同裁剪规则）调用 LLM 流式接口，逐段产出并写出答案文本，最后产出或附带引用列表

#### Scenario: 流式生成无匹配时的提示

- **WHEN** 调用方请求流式生成且检索返回空片段列表
- **THEN** 系统必须输出友好提示（不调用 LLM），并结束流，不编造内容

#### Scenario: 一次性生成行为在单轮下保持兼容

- **WHEN** 调用方使用现有一次性生成接口且无会话历史或无 `conversationId`
- **THEN** 系统行为须与引入多轮前单轮语义等价：返回完整 answer 与 citations，对外 JSON 结构可扩展字段但不得破坏原有字段含义
