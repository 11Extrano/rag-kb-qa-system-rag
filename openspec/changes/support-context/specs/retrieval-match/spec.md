## ADDED Requirements

### Requirement: 系统在向量检索前对查询进行改写（多轮场景）

当请求关联的会话中存在至少一轮历史消息时，系统必须在计算查询嵌入并执行 top-k 检索**之前**，调用 LLM 完成一次**改写**请求：改写输入须包含最近若干轮的 user 与 assistant 消息及本轮用户消息；改写输出须为**仅用于检索**的一句独立问句（或等价短文本），不得混入本轮 RAG 参考内容、不得承担最终用户可见答案的生成职责。改写与后续「作答」LLM 调用须为不同请求并使用不同 prompt；二者可共用同一 LLM 服务抽象。随后系统必须**仅**对该改写输出文本执行嵌入与向量检索。（MUST）

#### Scenario: 有历史时先改写再嵌入

- **WHEN** 会话中存在历史消息且用户提交了新一轮问题
- **THEN** 系统必须先得到改写输出的检索 query 文本，再对该文本调用 EmbeddingProvider 与向量存储检索，不得将会话全文直接拼接后作为嵌入输入

#### Scenario: 改写输入包含 assistant 消息

- **WHEN** 改写请求组装上下文时
- **THEN** 系统必须在输入中包含必要的 assistant 消息内容，以便消解依赖上一轮助手表述的指代；送去嵌入的文本仍为改写输出的单句

## MODIFIED Requirements

### Requirement: 系统接受用户的检索请求

系统必须接受检索请求；请求须关联本轮用户输入，并在多轮场景下关联 `conversationId` 以加载历史。在存在会话历史时，系统必须先执行「检索前改写」得到检索 query 文本（见 ADDED Requirement），再进入向量化；在无历史或约定退化为单轮时，改写输入可仅含本轮用户消息，嵌入仍针对改写输出文本（与单句等价时允许）。用于嵌入的检索 query 文本在改写成功后不得为空；若改写失败，系统必须按实现约定拒绝请求或采用文档化兜底（如使用本轮用户原文）并记录。（MUST）

#### Scenario: 用户以问题发起检索请求

- **WHEN** 用户通过指定接口提交问题且检索流程被触发
- **THEN** 系统必须将其作为检索流程的输入接受，并在多轮场景下先完成改写再执行嵌入检索

#### Scenario: 查询为空或缺失

- **WHEN** 本轮用户输入为空或缺失
- **THEN** 系统必须拒绝该请求并返回校验错误

#### Scenario: 无历史时会话仍有效

- **WHEN** `conversationId` 存在但该会话尚无历史消息
- **THEN** 系统必须仍能处理本轮问题，改写步骤退化后嵌入用的检索文本须与约定单轮行为一致

---

### Requirement: 系统按向量相似度匹配相关文档

系统必须将**检索 query 文本**（多轮场景下为改写步骤产出的独立问句）转为嵌入，通过与向量存储相同的 EmbeddingProvider，在 LanceDB 执行 top-k 相似度检索得到 chunk_id 与 score；再根据 chunk_id 从 MySQL 片段表取原文、关联文档表取文件名等，组装为「匹配到的内容」列表，并记录或发出「相关文档匹配完成」。匹配结果须按 score 降序排列。（MUST）

#### Scenario: 匹配返回带原文与来源的内容列表

- **WHEN** 已收到有效检索请求且向量库中已索引片段
- **THEN** 系统必须基于检索 query 文本计算查询嵌入、调用向量存储 top-k 检索得到 [{ chunk_id, score }, ...]，再根据 chunk_id 从 MySQL 取原文与文档元数据，组装为「匹配到的内容」列表（含 chunk_id、doc_id、原文 text、score、文档名等），按 score 降序，并标记「相关文档匹配完成」

#### Scenario: 向量库中无片段

- **WHEN** 向量库中没有任何片段（如尚未索引任何文档）
- **THEN** 系统必须返回空匹配列表，并可返回知识库为空的提示
