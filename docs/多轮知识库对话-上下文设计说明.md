# 多轮知识库对话与上下文：需求、问题与方案说明

本文档严格依据产品讨论与问答整理，说明「LLM 对接完成后如何支持上下文」相关目标、遇到的问题、解决思路与术语含义（精简）。

**阅读顺序**：先 **第 1 章 场景梳理**（弄清会在什么情况下出问题），再 **第 2 章 实现方案**（与场景一一对应的技术做法）。

---

## 1. 场景梳理

本章只回答：**在什么情况下**需要多轮、上下文、检索与模型两侧分别要扛什么；不展开具体接口与代码。

### 1.1 产品目标场景

在**知识库问答（RAG）**已能跑通的前提下，进一步要支持的局面：

| 场景 | 说明 |
|------|------|
| **多轮连续问答** | 同一会话内多轮 user / assistant，而不是每次只处理孤立的一句。 |
| **聊天可恢复** | 会话与消息持久化（例如 MySQL），刷新或下次打开仍能接着聊。 |
| **检索不能「只看当前一句」** | 用户常说短句、带指代（「海外呢？」「第二点呢？」），若检索只按当前句去向量匹配，容易**搜偏或搜空**。 |
| **多会话隔离、不做账号** | 用 **会话 id** 区分不同对话线；**不**做用户体系时，仍要避免 A、B 两场聊天串在一起。 |

---

### 1.2 会话与话题场景

| 场景 | 会发生什么 |
|------|------------|
| **用户想聊全新话题** | 若仍沿用旧会话，历史会混进检索与模型，答非所问或「翻旧账」。 |
| **同一会话内多轮** | 需要一条稳定的 **conversationId**，服务端按它取历史，且不与其它会话混淆。 |

要点：**新话题 = 新会话 id**；旧会话数据可以仍留在库里，但新请求只带新 id，检索与拼消息时**不读**旧会话。

---

### 1.3 消息持久化场景

| 场景 | 会发生什么 |
|------|------------|
| **用户基于上一轮助手内容追问** | 例如「展开第二点」「刚才说的费用」——若库里**没有** assistant 原文，服务端拼不出完整对话，检索融合与模型都缺半轮。 |
| **只存 user、不存 assistant** | 多轮 KB 场景下通常**不够**；惯例是 user 与 assistant **成对持久化**。 |

---

### 1.4 检索侧场景：短句、指代与向量化直觉

**典型局面**：第 1 轮问了较完整的问题并得到回答；第 2 轮用户只说「海外呢？」「费用呢？」——对话里听得懂，但**单独拿当前这句去做向量检索**时，文本太短、语义不完整，和知识库里**书面化、完整表述**的 chunk 不容易对齐。

**向量化在检索里的一步（直觉）**：

- 知识库 chunk 在**入库时**已经各自变成向量存在向量库里。
- **每一轮提问**，系统还要把某段「**检索用的文字**」变成**一个**查询向量，去和 chunk 向量比相似度、取 top-k。

```
检索 query 文本  ──embed()──▶  查询向量 q
chunk 已在库中的向量 v₁, v₂, …  ──比较相似度──▶  score，取 top-k
```

因此会出现：**检索 query 文本从哪来**直接决定搜得准不准。本文档**检索侧实现采用方案 C（改写再 embedding）**（见第 2.2 节）。下面用同一小例子固定直觉。

**场景设定（知识库里有什么）**  
假设某 chunk 大意是：「**海外订单须在签收后 7 日内发起退款申请，超时不再受理。**」

**多轮对话（第 2 轮很短）**

| 轮次 | 角色 | 内容（示意） |
|------|------|----------------|
| 1 | user | 国内订单退款要几天内申请？ |
| 1 | assistant | 国内订单需在签收后 15 日内……（略） |
| 2 | user | **海外呢？** |

第 2 轮「海外呢？」若**不经改写**就当作检索 query，容易对不上「海外订单退款期限」这类完整表述。采用方案 C 时，会先由**改写请求**得到独立问句（如「海外订单的退款申请期限是多久？」），再对该句做向量化检索；这与**作答请求**是两次调用（见第 2.2 节）。

---

### 1.5 模型上下文场景：长度、轮数与一次请求里有什么

| 场景 | 会发生什么 |
|------|------------|
| **只能带最近 N 轮** | 在模型上下文上限与成本下是**常规做法**；更早整轮不再进入本次请求，属于**滑动窗口**，是刻意设计，不是「坏了」。 |
| **同一条消息被从中间硬切断** | 容易语义残缺，应尽量避免；与「整轮丢弃」不同。 |
| **只限制轮数 N** | **不能保证**不爆上下文：每轮长短不一，还需要 **token 预算**（硬约束）与裁剪策略。 |
| **一次调用模型时要带什么** | 通常包括：**system**；可选 **经裁剪的多轮 user/assistant**；**本轮**的参考内容与用户问题。不会把整库向量发给模型，只有**本轮选中的 chunk 文本**。 |

---

### 1.6 存储结构场景（不区分用户时）

需要能按 **conversationId** 列出有序消息，并支持会话元数据（创建时间等）。**不区分用户**并不等于不要会话 id；没有会话 id 就无法隔离多场对话。

---

## 2. 实现方案

本章与 **第 1 章** 对应：先列**做法**，再指向更细的规则。

### 2.1 会话、话题与存储（对应 1.1～1.3、1.6）

| 做法 | 说明 |
|------|------|
| **会话 id** | 客户端或服务端生成并持久化；新话题使用**新** conversationId。 |
| **消息表** | `conversation_id`、`role`（user / assistant）、`content`、时间或序号；按会话查询排序。 |
| **会话表** | `id`、创建/更新时间；可选标题、metadata。 |
| **索引** | 按 `conversation_id` + 时间或序号，便于拉历史。 |

---

### 2.2 检索 query 与向量化（对应 1.4）

**已定**：检索侧采用 **方案 C：改写再 embedding**。

1. **改写请求**：输入为「最近若干轮对话 + 本轮用户消息」，输出为**仅用于检索**的一句独立问句（或等价短文本）；**不**承担生成最终答案、不混入 RAG 参考块。  
2. **向量化**：对改写得到的 **「检索 query 文本」** 调用 `embed` → `vectorStore.search`，得到 `MatchedContent[]`（与 `retrievalMatch` 中「对字符串做 embedding 再 search」一致）。

**先分清两层，避免混淆**：

| 环节 | 是否含 assistant 原文 | 说明 |
|------|------------------------|------|
| **改写模型的输入** | 通常**含**（user 与 assistant 交替的最近若干轮） | 给改写模型「看懂」指代、承接上一轮助手给出的信息（见下）。 |
| **送去 embedding 的文本** | **不含** | 只有**改写输出那一句**；不是把整段对话（含 AI 回复）拼起来做向量。 |

**改写输入里为何要带 assistant，而不是只给 user？**  
- **Embedding 只对一句**：最终参与向量检索的仍是**改写产出的独立问句**，不是把用户话和 AI 话拼成一大段去 embed。  
- **但改写模型需要读 assistant**：用户的短句常常指**上一轮助手刚说过的内容**。例如助手列了「一、二、三」三点，用户追问「第二点展开说下」——若改写输入里**没有** assistant 那段列表，模型无法把「第二点」展开成可检索的主题；仅保留 user 条会变成「第二点」缺主语。类似地，「刚才说的费用」「按你上面说的 B 方案」都依赖助手原文。  
- **仅 user 的简化**：若产品确认追问只依赖用户自己曾提过的主题、从不依赖助手表述，可将改写输入**收窄为仅最近若干条 user**（实现更简单），但会覆盖不全上述场景。

| 做法 | 检索 query 文本从哪来 | 说明 |
|------|------------------------|------|
| **C：改写再 embedding（采用）** | LLM 根据历史 + 当前句产出**一句独立检索问句**，再只对该句 embed | 指代消解通常更好；检索前**多一次 LLM 调用**（延迟与费用）。 |
| **A：仅用当前用户句** | 用户本轮输入原文 | 未采用；短句、指代多时易搜偏。 |
| **B：轻量拼接** | 最近若干轮 user/assistant 拼成一段 | 未采用；无额外 LLM，但长 assistant 可能带来向量**噪声**。 |

**改写与作答（生成答案）的关系**：二者可**共用同一 LLM 服务**（同一 `llmProvider`/同一 API 端点），但必须**分两次请求、使用不同 prompt/角色**，职责分离：

| 请求 | 职责 | 典型输入要点 |
|------|------|----------------|
| **改写** | 只产出「检索 query 文本」 | 历史 + 当前用户句；system 约束为「只输出可检索问句、不回答」等 |
| **作答** | 依据参考内容生成用户可见回答 | system + 历史 + 【参考内容】+ 用户问题；流式或非流式均可 |

**检索向量化**只对改写结果做 embed，用于**找 chunk**；**作答**在拿到 chunk 之后进行，二者顺序固定。一般**不**合并为「同一次请求既改写又依据参考内容作答」（除非产品另有约定；当前设计为**分开两次请求**）。

---

### 2.3 Token 预算与裁剪顺序（对应 1.5）

| 做法 | 说明 |
|------|------|
| **轮数 N + token 预算** | N 为策略上限；对「即将发送的完整输入」计数，不超过「模型上限 − 输出预留 − 安全余量」。二者同时用。 |
| **超预算时的裁剪顺序**（循环直至落入预算或达到下限） | ① **RAG 参考条**：优先去掉 **score 最低**的一条（本项目结果已按 score 降序，多为列表末尾）；可配合最少条数、单条长度截断。 ② **历史**：**成对**去掉最早一轮 user+assistant（滑动窗口）。 ③ **本轮 user 仍超长**：对该条截断或摘要。 ④ **最后**再动 system 中非关键句。 |

专业上可称：**上下文截断/裁剪**、**token 预算**、对历史为**滑动窗口**、对参考条为 **eviction** 等。

---

### 2.4 整体数据流（概念）

```
用户发送新消息（带 conversationId）
    → 从 DB 读取该会话历史（可先限制最多 N 轮）
    → 【LLM 改写】→ 检索 query 文本（方案 C，见 2.2）
    → embed(检索 query 文本) → 检索得到 MatchedContent[]
    → 组装 messages：system + 裁剪后的历史 + 本轮参考与用户问题
    → token 预算检查，不足则按第 2.3 节策略裁剪
    → 【LLM 作答】（流式或非流式；与改写不同请求、不同 prompt）
    → 将本轮 user 与 assistant 写入 DB
```

## 3. 名词与术语对照

| 名词 | 含义 |
|------|------|
| **多轮 KB / 多轮知识库对话** | 知识库问答形态下的多轮连续对话。 |
| **会话 id** | 标识一条对话线程；新话题通常对应新会话 id。 |
| **对话上下文** | 传给**生成答案用模型**的历史 user/assistant 消息序列。 |
| **检索上下文 / 检索 query** | 用于**向量检索**的那段文字；**已定**由「改写」步骤产出（方案 C），再 embed。 |
| **业务/会话元数据** | 如知识库范围、语言偏好等，与逐条聊天内容不同。 |
| **历史滑动窗口** | 只保留最近 N 轮（或等价 token 范围），更早整轮不进入本次请求。 |
| **Token 预算** | 单次请求输入不得超过的上限。 |
| **上下文截断 / 裁剪** | 为落入预算而减少 prompt 组成部分的统称。 |
| **最差的参考** | 当前保留的检索结果中 **score 最低**的 chunk（本项目已按 score 降序）。 |
| **user 仍超长** | 本轮用户单条输入过长，需对该条单独截断或摘要。 |
